UI 적용 가능성추상화 개념은 있으나 UI 속성으로는 덜 명시됨UI 상태, 조건, 속성, 피드백 구조로 변환빠뜨려서 생긴 차이
  • 분해단계
    • 고수준 목적 관점차이
      • 고수준의 목적은 측정과 사용자가 인식 가능하게 표시 하는 것을 추가하면 완성도가 더 높아짐
    • 중수준 구조 관점차이
      • 항목을 기능으로 그룹핑 하면 문서 퀄이 올라간다
    • 저수준 요소 빠뜨림
      • 저수준 요소에는 문장 뿐 아니라 숫자나 단위로 측정 가능한 값이 들어 가야한다.
    • 고수준은 목적, 중수준은 구조. 저수준은 요소 로 생각하면 명확하다.
  • 패턴인식 단계
    • 동력원 관점차이
      • 나는 물리적 입출력을, AI는 상태 변화와 UI 피드백을 보았다.
    • 입력/출력 관점차이
      • 이 둘을 연결해 입력 → 상태변화 → 출력 표시로 정리가 가능 하다
    • 상태제어 빠뜨림
      • 나의 상태제어의 아이디어는 좋지만 플로우 조건으로는 연결성이 부족하다.
        • if/else 조건으로 구체화 해야 한다.
    • 변수관계 빠뜨림
      • 변수 목록을 관계 문장으로 변경하면 패턴인식이 더 강해진다.
  • 추상화 단계
    • 핵심모델 관점차이
      • 나의 모델은 매우 간결하며 시각적 프로토타입에 적합하다. AI 모델은 디지털 UI 설계로 확장하기 좋다.
    • 물리요소 변환 관점차이
      • 사용자 모델은 “에너지”의 은유가 강하고, AI 모델은 구현 가능한 속성명이 강하다. 프로젝트 목적이 UI 라면 AI 식 프로퍼티가 더 유리.
    • 방향성 관점차이
      • 나는 공간적 방향을 다양하게 실험해 좋은 확장성을 보임. AI는 방향을 제거해 더 일반화 함
    • 상태 빠뜨림 관점차이
      • 사용자의 상태는 핵심만 있다.
        • UI플로우차트 에서는 대기와 오류같은 보조 상태가 필요하다.
  • 플로우 차트 단계
    • 기본 흐름관점차이
      • 나의 흐름은 원형 반복 구조를 잘 잡았다.
      • AI는 시스템이 실행될 때 필요한 판단 단계를 더 세분화 했다.
    • 조건분기 빠뜨림
      • 플로우차트의 유의사항에 맞추려면 조건을 명확한 if/else 문장으로 바꾸는 작업이 필요하다.
    • 상태제어빠뜨림
      • 멈춤이 왜 발생하는지, 멈춘 뒤 어떤 선택지가 있는지 명시하면 더 설계가 가능한 플로우가 된다.
    • 프로토타입 연결 관점차이
      • 나는 일관성이 좋다. 다음단계에서는 UI 화면 요소와 상호작용 조건을 연결 해야 한다.

Takeaway

  • 요소를 기능 별로 그룹핑 해보기
  • 특정 가능한 단위까지 쪼개 보기
  • 상태를 명확하게 정의하고
  • 조건을 명확하게 정의하기 (if/else)

“멈춤”의 원인이 모호하다빠뜨려서 생긴 차이사용자가 멈춘 것인지, 시스템 오류인지, 자원이 부족한 것인지 구분해야 한다
UI 상태가 덜 세분화되어 있다빠뜨려서 생긴 차이인터페이스 설계에서는 대기, 작동, 일시정지, 완료, 오류를 분리하면 화면 설계가 쉬워진다

가장 중요한 차이 5가지

번호차이차이 유형의미
1사용자는 물리적 특징을 잘 관찰했지만, 시스템 단위로 묶는 구조화가 약했다빠뜨려서 생긴 차이관찰에서 시스템 설계로 넘어가는 연결이 필요하다
2패턴 인식에서 요소는 찾았지만, 변수 간 관계가 덜 드러났다빠뜨려서 생긴 차이“무엇이 있다”보다 “무엇이 변하면 무엇이 변한다”가 중요하다
3에너지 추상화는 좋지만, UI 적용에는 자원·상태·피드백 개념이 더 유리하다관점이 달라서 생긴 차이에너지는 물리적이고, 자원은 디지털 시스템까지 확장 가능하다
4제어를 “멈춤” 중심으로 보았지만, UI에서는 시작·중지·재개·완료·오류까지 제어에 포함된다몰라서 생긴 차이상태 제어의 범위를 넓게 이해할 필요가 있다
5플로우차트가 순환 구조는 좋지만 If/Else 조건이 부족했다빠뜨려서 생긴 차이조건 분기를 넣으면 실제 인터페이스 설계에 바로 사용할 수 있다

사용자 시도의 강점

강점설명
핵심 구조를 빠르게 잡았다“A에서 B로 이동한다”는 구조는 추상화의 중심이 될 수 있다
물리적 원인을 잘 관찰했다중력, 구멍, 모래량, 투명성, 대칭성을 잘 포착했다
추상화 방향이 명확했다모래를 에너지로 바꾸며 물리적 실체에서 벗어나려는 시도가 있었다
플로우차트가 간결했다시작, 완료, 다시시작의 큰 흐름이 잘 보인다
질문형 메모가 좋다“왜 대칭이어야 하는가?”, “곡선 형태가 많은 이유?” 같은 질문은 분석을 더 깊게 만든다

개선하면 좋은 지점

개선 지점보완 방법
분해 단계요소를 기능별 시스템으로 묶기
저수준 단계관찰 표현을 측정 가능한 단위로 바꾸기
패턴 인식변수 간 관계를 화살표나 조건문으로 표현하기
추상화에너지뿐 아니라 자원, 상태, 전환, 피드백 개념 추가하기
플로우차트If/Else 조건과 상태 전이를 명확히 쓰기

UI 시스템용 보완 모델

속성의미
