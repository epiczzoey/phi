HID 수업 4주차 1pager

문제정의

  • LLM 채팅에서 수정 요청 후 검수 비용이 사용자에게 되돌아옴
  • 수정 답변은 새 텍스트 덩어리로 교체 → 무엇이 바뀌었는지 사용자가 직접 대조

사용자

  • 누구: LLM에게 실제 결과물(기획안·문서·설계)을 맡기는 실무자
  • 사용 목적: 잡담이 아닌 output 생산 — 한 번에 완성되지 않고 반복 수정으로 완성
  • 기대 경험: 맡겼으면 알아서 반영 → 나는 결과만 빠르게 확인

페인포인트

  1. 검수가 다시 생김 — 수정 후에도 이전 답변과 새 답변을 스스로 비교해야 함
  1. 맥락 반복 설명 — "2단계만 바꿔줘" 같은 이전 요청 맥락을 매 수정마다 재입력
  1. 상태 불명확 — 입력창이 항상 비어있어 지금 수정 중인지 새 요청인지 인터페이스가 말해주지 않음
  1. diff 정보 과부하 — 변경 표시가 항상 노출되면 최종 결과만 보고 싶은 사용자에게 노이즈

사용자의 목표

  • 검수하지 않는다 — 결과물이 무엇이 어떻게 바뀌었는지 스스로 보여주기
  • 한 번만 설명한다 — 수정 요청이 이전 맥락에서 이어지기
  • 상태를 먼저 안다 — 지금 수정 중임을 인터페이스가 먼저 알려주기
  • 밀도를 선택한다 — 변경 상세 vs 최종 결과만, 둘 중 선택 가능

핵심마찰 정의

  1. 검수 마찰 — "뭐가 바뀌었는지 모른다"
    • 수정 답변 = 기존 답변 통째 교체
    • 사용자가 이전 답변 기억하거나 스크롤해서 직접 대조
    • → 검수 비용이 사용자에게 고스란히 전가
  1. 통제 마찰 — "수정이 새 대화처럼 느껴진다"
    • 입력창 항상 비어있음 → 이전 요청과의 연결 단절
    • 수정 의도를 처음부터 재입력해야 함
    • 수정 중이라는 상태 표시 없음
    • → 수정이 흐름이 아니라 새 작업이 됨

인터페이스 디벨롭

편집 모드 → 통제 마찰 해소

  • 편집 진입 방식
    • 답변 카드 하단 연필 아이콘 클릭 → 직전 요청 텍스트가 입력창에 자동 삽입
    • 처음부터 다시 쓰지 않고 필요한 부분만 수정 후 전송
  • 자동 포커스 + 커서 끝 이동
    • 아이콘 클릭 즉시: 포커스 획득 + 커서가 텍스트 맨 끝으로 위치
    • 인터페이스가 먼저 "바로 타이핑하세요"를 행동으로 전달
  • 힌트 텍스트 전환
    • 기본: "⌘ Enter로 전송" → 편집 중: "수정 후 ⌘ Enter로 전송"
    • 같은 위치, 같은 크기 — 텍스트 교체만으로 모드 전환을 전달

answer-diff 카드 → 검수 마찰 해소

  • 카드 생성 방식
    • 편집 모드 전송 시: 1차 답변(answer-doc)이 비활성화, 0.4초 후 answer-diff 카드 생성
    • 새 텍스트가 아닌 머지형 diff — 원본 전체 보존 + 변경된 줄만 인라인 강조
  • 헤더 변경 카운트 칩
    • 초록 칩: + N줄 추가 / 빨강 칩: − N줄 삭제
    • 읽기 전에 변경 규모 파악 가능 — 칩 색상이 성격(추가/삭제)을 직관적으로 전달
  • 인라인 diff 강조
    • 추가된 줄: + 기호 + 초록 배경 + 초록 좌측 보더
    • 삭제된 줄: − 기호 + 연한 빨강 배경 + 취소선 + 흐린 텍스트
    • 변경 없는 줄: 기호 없음 + 기본 본문색
    • 시선이 강조 색상으로 자연스럽게 변경 지점으로 이동
  • 하단 범례
    • + 추가됨 / − 삭제됨 / ● 변경 없음 — 3가지 기호 설명
    • diff 모드에서만 표시 (clean 모드 전환 시 함께 숨겨짐)
  • 카드 스타일 일관성
    • answer-doc과 동일: 흰 배경 + 1px 보더 + radius 12px + 하단 액션 바
    • 형식이 다른 두 카드가 같은 시스템에 속한다는 시각적 일관성 유지

diff 토글 → 정보 밀도를 사용자가 선택

  • 헤더 우측 토글 버튼 — "변경사항 숨기기 / 변경사항 보기"
  • diff 모드 (기본)
    • +/− 기호, 색상 강조, 범례 전체 표시
    • 변경사항을 정확히 파악하려는 사용자용
  • clean 모드 (토글 후)
    • 삭제된 줄 숨김 → 추가 + 유지된 줄만 깔끔하게 표시
    • 범례도 함께 사라짐 — diff 관련 UI 전체가 일관되게 숨겨짐
    • 최종 결과만 보거나 복사하려는 사용자용
  • 언제든 "변경사항 보기"로 diff 모드 복귀 가능

2종 카드 시스템 → 카드 자체가 맥락을 전달

  • answer-doc — 1차 답변 카드
    • 노션식 문서 포맷: 콜아웃 · 구분선 · 굵게 · 헤딩
    • 우측 점프 네비게이션: 섹션 제목 클릭 → 해당 위치 스크롤
  • answer-diff — 2차 수정 결과 카드
    • 머지형 diff 포맷: 변경 카운트 칩 + 토글 + diff/clean 2모드
    • 카드 종류만으로 "처음 생성" vs "수정 결과" 구분 가능
  • 1차 답변은 2차 전송 시 자동 비활성화 (frozen) → "이전 결과물"임을 시각적으로 표시

시나리오

단계사용자 행동화면인터랙션해소 마찰
1요청 입력 후 전송하단 입력창"⌘ Enter로 전송" 힌트사용자 버블 생성→ 0.4초 후 answer-doc 카드 등장→ 자동 스크롤—
2답변 읽기노션식 문서 카드(콜아웃·구분선·굵게)우측 점프 네비 → 원하는 섹션 바로 이동스크롤 압박
3일부 수정 원함카드 하단 연필 아이콘(hover 시 노출)클릭 → 직전 요청 자동 삽입커서 텍스트 끝 이동입력창 테두리 → 검정통제 마찰(반복 설명)
4편집 배너 확인입력창 위 회색 배너(연필 + 안내 + ✕)"수정 중" 상태 명시힌트 → "수정 후 ⌘ Enter로 전송"통제 마찰(상태 불명확)
5필요한 부분만 수정 후 전송—1차 answer-doc → 비활성(frozen)0.4초 후 answer-diff 카드 생성—
6무엇이 바뀌었는지 확인answer-diff 헤더"+ N줄 추가 · − N줄 삭제" 칩+ 초록 배경 행− 빨강 취소선 행하단 범례 3종검수 마찰
7diff가 부담됨헤더 "변경사항 숨기기" 버튼클릭 → clean 모드삭제 줄 사라짐, 범례도 숨겨짐"변경사항 보기"로 복귀 가능정보 과부하
8수정 취소편집 배너 ✕ 버튼편집 해제, 입력창 초기화테두리 → 회색 복귀—
9결과 복사카드 하단 복사 아이콘삭제 줄 제외 평문 클립보드 복사1.2초간 ✓ 아이콘 전환—
