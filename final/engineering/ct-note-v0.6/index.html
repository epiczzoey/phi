STEP 1

분해 (Decomposition)

사물: 습관 씨앗 정원 앱 하나를 고수준 → 중수준(하위 시스템) → 저수준(수치화 요소)으로 나눈다.

"사용자가 습관을 씨앗으로 등록하고, 매일의 실행을 알림 체크인으로 기록하여, 그 누적된 돌봄이 정원 속 식물의 성장으로 가시화되는 시스템"

트리 구조

  B["B. 알림 체크인 시스템

STEP 2

패턴 인식 (Pattern Recognition)

사물이 하나뿐이므로, 앱을 구성하는 세 하위 시스템(A/B/C)을 나란히 놓고 공통 규칙과 분기점을 찾는다.

비교 속성A. 씨앗 심기B. 알림 체크인C. 성장·생기 표현
트리거 발생 위치앱 내부 (능동)앱 외부 (OS 알림, 수동 도달)트리거 없음 (파생값)
입력 형태텍스트 + 시간 선택단일 버튼 탭없음 (계산됨)
출력 형태정원에 씨앗 등장오늘 로그 저장식물 형태·색 변화
상태 지속성1회성 (생성 시 한 번)매일 반복누적·파생 (영구 반영)
공통 규칙모든 하위 시스템은 "트리거 발생 → 상태값 갱신 → 시각 피드백"의 3단 구조를 공유한다.
분기점트리거가 앱 내부 / 외부 / 파생 중 어디서 발생하는지에 따라 시스템의 반응 방식이 갈린다.
STEP 3

추상화 (Abstraction)

"특정 트리거(등록/체크인)에 의해 누적되는 '돌봄 값'을 규칙에 따라 두 개의 파생 상태(성장 단계, 생기 수준)로 변환해 시각적으로 표현하는 시스템"

변수 / 상수

구분이름설명
변수careEvent등록 | 체크인
변수totalCare누적 돌봄 횟수
변수recentCare최근 7일 돌봄 횟수
상수GROWTH_THRESHOLDS[3, 7, 14, 30]
상수VITALITY_WINDOW7일 (기획서 recentLogs7Days 기준)

이벤트 (if → then)

조건 (IF)결과 (THEN)
careEvent == 등록Habit 생성, growthStage = 0
careEvent == 체크인 AND 오늘 미기록Log 생성, totalCare += 1
totalCare가 임계값을 넘음growthStage += 1 (역행 없음)
(항상)vitalityLevel = recentCare / 7 * 100
은유적 대응growthStage = 누적된 노력(사용자의 성장) · vitalityLevel = 최근 컨디션

순서도 기호 범례

아래 전체 프로세스 다이어그램은 KS/ISO 표준 순서도 기호 6종만 사용해 그렸다. 기호가 곧 단계의 성격(시작/끝·준비·처리·판단·입출력·반복)을 나타내므로, 각 도형 안에는 구체적인 동작만 적는다. 개발할 때 익숙한 용어(Event · Activity · If/Then · Input · Output)도 함께 표시해, 순서도 기호와 실제 코드 개념을 바로 연결할 수 있게 했다.

전체 프로세스 순서도

처음과 끝을 추상적인 이벤트가 아니라 실제 Input / Output으로 설계했다. 처음은 "사용자가 습관 정보를 입력하거나 물을 줬다고 탭하는 순간"(Input)이고, 끝은 "정원 화면이 갱신되어 최종적으로 보이는 순간"(Output) — 성공 사이클이면 성장한 정원, 초기화 사이클이면 빈 정원이다. "다음 알림 시각 계산"은 그 값이 오직 바로 다음 반복 조건에만 쓰이므로, 별도 처리 박스로 두지 않고 반복(Loop) 도형의 조건부에 합쳐 그렸다. 오른쪽 바깥 통로들은 모두 다시 이 반복 박스로 합류해 하루 단위 반복을 재시작한다. v0.6 추가: Output 직전에 "만개한 식물이 6개 이상인가요?" 판단을 넣어, 정원 배치가 그리드에서 원형 다이스로 전환되는 분기를 반영했다.

발견된 보완 필요 지점다이어그램은 "나중에"를 탭하거나 이미 기록된 경우에도 다음 알림 시각을 다시 계산한 뒤 대기하도록(의도된 설계) 그렸다. 그런데 실제 코드의 mockLater 핸들러는 배너만 닫고 scheduleNextNotification()을 다시 호출하지 않는다 — 즉 사용자가 다른 동작을 하기 전까지는 알림이 자동으로 재예약되지 않는 구현상의 틈이 있다. 이 문서의 순서도가 의도된 설계이므로, 프로토타입 쪽 코드를 이 설계에 맞춰 보완하는 것을 다음 작업으로 남겨둔다.
도형 통합"다음 알림 시각 계산"(처리)과 "반복 조건 · 알림 시각 도달"(반복)은 항상 붙어서 1:1로만 이어지고, 계산 결과가 오직 바로 다음 조건 판단에만 쓰인다 — 즉 독자적인 의미가 없는 부속 계산이다. 그래서 별도 처리 박스로 두지 않고 반복 도형의 조건부 안에 함께 적어 한 박스로 합쳤다. 범례의 "준비" 기호는 표준 순서도 6종 설명을 위해 남겨두지만, 이 프로세스에는 별도 도형으로 등장하지 않는다.
실행 순서 조정"vitality 갱신"(레벨 갱신)과 "성장 임계값 초과? → growthStage +1"은 서로 참조하는 값이 없는 독립적인 계산이다. 그래서 vitality 갱신을 판단보다 먼저 실행해 "레벨 갱신은 성장 여부와 상관없이 항상 일어난다"는 것을 시각적으로 분명히 하고, growthStage +1은 조건이 맞을 때만 실행된 뒤 같은 지점(Output 직전)에서 합류하도록 순서를 바꿨다. 두 순서 모두 최종 상태값은 동일하다.
